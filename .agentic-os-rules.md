# Agentic OS V4 — AI Development Rules & Coding Standards

> **Purpose:** This file defines strict rules for AI coding assistants when modifying
> the Agentic OS V4 codebase. It prevents common mistakes, ensures consistency,
> and maintains the integrity of the merged codebase.

---

## 🏛️ ARCHITECTURE RULES

### AR-001: Core Must Be Rust
- All performance-critical paths MUST be implemented in Rust
- TypeScript is ONLY for: skills, plugins, recipes, dashboard UI, desktop UI
- Provider adapters MAY be in TS (dynamic configuration) or Rust (performance)
- **Exception:** Python and Go code from source projects MUST be rewritten (not ported line-by-line)

### AR-002: ACP Is The Unification Protocol
- All service-to-service communication MUST use ACP (Agent Client Protocol)
- No direct HTTP/REST calls between internal components
- External-facing APIs MAY use REST/gRPC but MUST be wrapped in ACP-compatible facade

### AR-003: Config Is TOML-First
- All configuration MUST be authored in TOML
- YAML/JSON/Env config MUST be converted to TOML internal representation
- Schema validation MUST use JSON Schema (auto-generated from Rust types)
- Config changes MUST be validated before application

### AR-004: Single Source of Truth for Types
- Core types MUST be defined in Rust (`crates/core/src/types/`)
- TypeScript types MUST be auto-generated from Rust types via `ts-rs` or similar
- No duplicate type definitions — this caused the fragmentation in source projects

### AR-005: Feature Flags for All Breaking Changes
- Every breaking change MUST have a feature flag
- Feature flags MUST be removable after 2 releases
- Migration warnings MUST be logged when deprecated features are used

## 📝 CODING STANDARDS

### CS-001: Error Handling
```rust
// ✅ GOOD: Use this crate's error types consistently
#[derive(Debug, thiserror::Error)]
pub enum AgenticError {
    #[error("Provider '{provider}' returned {status}: {message}")]
    ProviderError { provider: String, status: u16, message: String },
    
    #[error("Authentication failed: {details}")]
    AuthError { details: String },
    
    #[error("Rate limited. Retry after {retry_after}s")]
    RateLimited { retry_after: f64, provider: String },
    
    #[error("Routing failed: no available providers for model '{model}'")]
    NoAvailableProviders { model: String },
    
    #[error("Config validation failed: {errors}")]
    ConfigError { errors: Vec<String> },
}

// ❌ BAD: Generic errors
fn do_something() -> Result<(), Box<dyn Error>> { }
fn do_something() -> Result<(), String> { }
```

### CS-002: Logging
```rust
// ✅ GOOD: Structured logging with context
tracing::info!(
    provider = %provider_id,
    model = %model_id,
    latency_ms = latency.as_millis(),
    "Provider request completed"
);

// ❌ BAD: Unstructured logging without context
tracing::info!("Provider request completed");
```

### CS-003: Async Patterns
```rust
// ✅ GOOD: Always use tokio for async runtime
#[async_trait]
pub trait ProviderAdapter: Send + Sync {
    async fn chat(&self, req: ChatRequest) -> Result<ChatResponse>;
}

// ✅ GOOD: Use bounded channels for streaming
use tokio::sync::mpsc;
let (tx, rx) = mpsc::channel::<StreamChunk>(128);

// ❌ BAD: Unbounded channels
let (tx, rx) = tokio::sync::mpsc::unbounded_channel();
```

### CS-004: Testing Requirements
```rust
// ✅ GOOD: Every public function must have tests
#[cfg(test)]
mod tests {
    use super::*;
    
    #[tokio::test]
    async fn test_provider_chat_basic() {
        let provider = MockProvider::new();
        let req = ChatRequest::default();
        let resp = provider.chat(req).await.unwrap();
        assert!(resp.is_success());
    }
    
    #[tokio::test]
    async fn test_provider_chat_retry_on_429() {
        let provider = MockProvider::new()
            .with_rate_limit_response();
        let req = ChatRequest::default();
        // Should retry and succeed
        let resp = provider.chat(req).await.unwrap();
    }
}
```

### CS-005: Documentation Requirements
- Every public API MUST have doc comments
- All config options MUST have documentation
- Breaking changes MUST have migration guide entries
- Each module MUST have a module-level doc comment

## 🔒 SECURITY RULES

### SR-001: No Secrets in Code
- API keys, tokens, passwords MUST NOT be hardcoded
- Always use env var references: `api_key = "${OPENAI_API_KEY}"`
- Credential store MUST use OS keychain where possible

### SR-002: Input Validation
- All external input MUST be validated
- Provider responses MUST be validated before passing to caller
- Use `Validated<T>` wrapper types for parsed config

### SR-003: Sandbox Enforcement
- Skills from untrusted sources MUST run in WASM sandbox
- Shell commands MUST be confirmed through consent bus
- File system access MUST follow policy engine rules

### SR-004: Audit Trail
- All auth decisions MUST be logged
- All config changes MUST be logged (before/after)
- All provider requests/responses CAN be logged (configurable)

## 🧪 INTEGRATION RULES

### IR-001: Copy-Then-Surgically-Edit
When importing code from source projects:
1. Copy the file exactly (with license header preserved)
2. Add a comment: `// Imported from: <project> SHA:<commit-hash>`
3. Make minimal surgical edits to adapt to unified interfaces
4. Do NOT rewrite working code — this causes regressions

### IR-002: Keep Original Tests
- All tests from source projects MUST be preserved
- New tests MUST be added for unified functionality
- Test files from different projects MUST be kept separate during merge

### IR-003: No Dead Code Accumulation
- Config options from source projects that are deprecated MUST be logged as warnings
- Old code paths MUST be removed after 2 releases with migration guides
- Feature flags MUST have removal deadlines

## 🚀 WORKFLOW RULES

### WR-001: One PR Per Phase
- Each subphase MUST be a separate PR
- PR description MUST list:
  - Source project files copied
  - Edits made
  - Acceptance criteria met
  - Developer's note (what to watch for)

### WR-002: PR Size Limit
- Max 10 files changed per PR
- Max 500 lines added per PR (excludes copied files with preserved licenses)
- Larger changes MUST be broken into multiple PRs

### WR-003: Review Checklist
- [ ] All source project licenses preserved
- [ ] No secrets in code
- [ ] Tests pass (unit + integration)
- [ ] No breaking changes without feature flag
- [ ] Config changes validated
- [ ] Documentation updated
- [ ] Changelog entry added

## ⚡ PERFORMANCE RULES

### PR-001: No Blocking in Async
- All I/O MUST be async
- CPU-intensive work MUST be on blocking threads or via `spawn_blocking`
- Never call `.await` on something that should be immediate

### PR-002: Cache Aggressively
- Provider model lists MUST be cached (TTL: 5 min)
- Config parsing MUST be cached (invalidate on file change)
- Provider health checks MUST be cached (TTL: 30s)

### PR-003: Profile Before Optimizing
- No optimization without benchmarks
- Benchmarks MUST be added before optimization
- Results MUST show >20% improvement to justify change

## 🔄 MIGRATION SAFETY RULES

### MSR-001: Never Delete Without Deprecation
1. Add deprecation warning in version N
2. Keep functionality through version N+1
3. Remove in version N+2
4. Migration guide MUST exist for all 3 steps

### MSR-002: Config Migration Is Read-Only
- Migration from old formats NEVER modifies source file
- Output is always to new config location
- Source file path is preserved for rollback
- Backup is created before any config write

### MSR-003: Database Migrations Are Reversible
- All schema migrations MUST have up AND down scripts
- Migrations MUST be idempotent
- Rollback plan MUST exist for every migration
- Test rollback in CI

## Initial findings and remediation

### [P0] Documented frozen install path was not reproducible — resolved

**Initial evidence:** root `package.json` had no dependency keys while the root importer in `pnpm-lock.yaml` declared the frontend toolchain; the server manifest also differed from its lockfile importer.

**Remediation:** restored the root dependency/devDependency declarations, added the missing workspace/server entries represented by the lockfile, and regenerated the lockfile. `corepack pnpm install --frozen-lockfile --ignore-scripts` now reports the lockfile is up to date and links all workspace packages.

**Remaining caveat:** a full lifecycle install is still blocked by the environment's inability to download Node headers for `better-sqlite3` due TLS/network failure. This is documented as an environment blocker, not a pass.

### [P1] `npx` made frontend validation non-deterministic — resolved

**Initial evidence:** root scripts used `npx vite` and `npx vite build`; the baseline attempted a temporary Vite 8 install even though the lockfile pins Vite 5.

**Remediation:** root scripts now call declared local `vite` binaries. `npm run build:frontend` completes successfully using Vite 5.4.21. One CSS minifier warning remains and is recorded in the baseline.

### [P1] Baseline used npm rather than the declared workspace path — partially resolved/documented

The documented install path is now the root pnpm workspace. The existing scripts still shell into packages using npm, so the validation command structure remains mixed. It returns deterministic non-zero codes, but a future CI cleanup should use one explicit workspace runner consistently.

### [P0] Root `tsconfig.json` was invalid JSON — resolved

The file contained two adjacent JSON objects. This blocked the frontend build and test runner before source validation. The duplicate server configuration object was removed; the server already has its own `server/tsconfig.json`. The frontend build now loads the config successfully.

## Current blocking findings

### [P1] Full install is not reproducible on this environment because `better-sqlite3` cannot build

`corepack pnpm install --frozen-lockfile` reaches the lifecycle step, but the `better-sqlite3` prebuilt download cannot verify TLS and the fallback build cannot download Node headers. `--ignore-scripts` links packages but leaves the native binding absent.

**Required follow-up:** run the full install in CI/clean environment with working certificate/network access or provide a documented compatible prebuilt/build-tool path. Tests must run with the native binding present.

### [P1] Repository lint, typecheck, and test suites are not green

- `npm run lint`: 536 errors and 90 warnings, including server environment globals, undefined symbols, unused variables, and explicit-any warnings.
- `npm run typecheck`: existing errors across env/config/service contracts, missing modules, implicit anys, and undefined symbols.
- `npm test`: native SQLite binding failures, missing module/test failures, and a Node out-of-memory termination.

These are outside the scope of a validation-baseline documentation story, but they are release blockers and must remain visible in sprint tracking.

### [P2] The root build still emits a CSS minifier warning

The production build succeeds, but the minifier reports an invalid `file:line` CSS property generated by the current styling pipeline. This should be triaged before treating the frontend build as warning-free.

## What is good

- Baseline commands were attempted without suppressing exit codes.
- Missing toolchain failures were recorded rather than treated as passes.
- Manifest/lockfile drift is now surfaced and the frozen lockfile resolution passes.
- Frontend scripts no longer reach the network for a moving tool version.
- The invalid root TypeScript configuration was found through validation and corrected.
- The baseline clearly distinguishes static install success from lifecycle/native build failure.
- The story warns against `|| true`, silent catch blocks, and server-only lockfiles.

## Review conclusion

The initial reproducibility defects found by this review were corrected, but the story cannot be marked `done` while the dependency-backed validation baseline remains blocked and repository lint/typecheck/tests remain failing. Keep E0-S1 in `blocked` status, link the remediation evidence, and open follow-up stories for native dependency CI, TypeScript/lint repair, and test-suite stabilization.

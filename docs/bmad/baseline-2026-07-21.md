## Installation guidance

The repository declares a pnpm workspace with a committed `pnpm-lock.yaml` using lockfile version 9. The intended clean install is:

```bash
corepack enable
pnpm install --frozen-lockfile
```

However, this command was not executable in the baseline because pnpm is absent, and the tracked root `package.json` currently has no `dependencies` or `devDependencies` keys while the root importer in `pnpm-lock.yaml` contains them. The lockfile/manifest relationship must be reconciled and then verified before calling the install path reproducible.

If Corepack is unavailable, install a pnpm version compatible with lockfile version 9, then run the same command from the repository root. Do not generate a server-only `package-lock.json` as a substitute for the workspace lockfile.

The development guide was updated to remove the previous ambiguity between root workspace installation and server-only npm installation, but it is not yet a green install guarantee.

## Existing validation entry points

The root `package.json` already exposes deterministic chained commands:

- `npm run lint`

- `npm test`

- `npm run build:frontend`

The server package exposes equivalent `lint`, `typecheck`, `test`, and `validate` scripts. These commands fail fast and return non-zero exit codes when tools are unavailable. The R1 implementation should retain this behavior and add dependency installation/CI verification rather than hide the failures.
